# 科学编程人机协作协议

> 适用：AI Agent 协助的科学计算 / 仿真 / 数据处理 / 领域专用软件
> 目标：让 AI 知道在科学项目里**该做什么、该怎么做、绝不能做什么**

---

## 协议总览：三件事

1. **正确性有锚点**——所有"对不对"必须对照一个权威源，而不是 AI 自己说对
2. **解释有据**——任何被引入的参数、公式、修正，必须能说出"它来自哪里、为什么它合理"
3. **人在场**——AI 不做架构判断、不做物理判断、不做"对的原因"判断；这些事留给人

---

## 0. 协议启动检查（AI 接到任务时先问自己）

接到任何科学编程任务，先做以下 4 项检查。**任一项不满足，先向用户确认再开工。**

- [ ] **有权威参考源吗？**（论文里的参考实现 / 教科书公式 / 实验测量数据 / 行业标准）
- [ ] **有明确的误差阈值吗？**（如"对照参考实现误差 ≤1%"）
- [ ] **有基准测试点 + 变动测试点吗？**（如"Planck 2018 基准 + 2 个变动宇宙学"）
- [ ] **有领域专家能在卡住时被叫到吗？**

如果用户没给这些信息，**先问**。不要直接开工。

---

## 1. AI 必须做的事（MUST）

### 1.1 测试先于代码

接到一个新函数 / 新模块的实现任务，**先写对照测试，再写实现**。

- 找出对应的参考输出（从权威源生成，或从已验证数据中提取）
- 写一个对比函数：input → 你的输出 vs 参考输出 → 误差
- 跑测试（应当失败）→ 写实现 → 跑测试（应当通过）

**为什么**：让 AI 在产出前就知道"对的输出长什么样"，避免"先写后改"。

### 1.2 记录一切决策

每次会话的关键决策，写到 CHANGELOG / session log：

- 尝试过什么、为什么尝试
- 失败了什么、为什么失败
- 关键架构决策的"为什么"
- 已知死胡同（避免下次再走）

**格式建议**：
```
## Session [N] - [日期]
**目标**: ...
**尝试**: ...
**结果**: ...
**结论**: [为什么这样设计 / 下次别再走这条路]
```

### 1.3 任何数值修改都要给出依据

- 修改一个系数 → 引用参考实现的行号 / 公式来源
- 引入一个新参数 → 说明这个参数在参考实现中**出现过**、有**物理推导**
- 改一个算法 → 解释为什么这个算法比原来的更适用

**禁止**：写"我加了一个修正系数让测试通过"——这是凑系数，是协议明确禁止的。

### 1.4 主动报告"我做不到"

以下情况，**立即向用户报告，不要硬撑**：

- 同一个方向连续 5 次修改没有单调改进
- 引入新参数后，关闭该参数（设为 0）反而让误差爆
- 某个 fix 让一个指标改善的同时让另一个指标变差
- 不知道某个参数在物理上对应什么
- 对当前架构是否对越来越不确定

**报告格式**：
```
⚠️ 异常信号
- 现象: [...]
- 我的判断: [...]
- 需要你做的判断: [架构重设计 / 物理判断 / 接受现状]
```

### 1.5 在每个提交前跑完整测试

提交任何 fix / 新代码 / 重构，**必须**：

- 在基准测试点通过
- 在所有变动测试点通过
- 跑一次"极限探针"：把这一轮引入的所有新参数设为 0 / 边界值，确认误差不会爆
- 把测试结果写进 CHANGELOG

---

## 2. AI 应该做的事（SHOULD）

### 2.1 在做之前先想清楚"我能不能做"

接到任务时，自评：

- ✅ 能做：对照参考实现转录、定位数值差异、修正系数、跑测试、debug 单点 bug
- ⚠️ 需要人加速：量级异常、维度不匹配、需要领域知识判断"哪个方向值得试"
- ❌ 自己做不了：架构是否与物理兼容、"这个解为什么对"的物理解释、识别凑系数

**自评结果如果是 ❌，立即升级，不要硬上。**

### 2.2 把测试输出控制得简洁

- 测试成功输出：≤ 10 行
- 测试失败输出：≤ 20 行
- 详细诊断写到日志文件，不污染输出

**为什么**：啰嗦的输出吞噬上下文窗口，让 AI 看不到真正的问题。

### 2.3 主动质疑自己的"完美解"

每次给出"通过所有测试"的方案，**自问 3 个问题**：

1. 这个解"对"，是因为它真的对，还是因为它被校准到测试点？
2. 如果改一个测试参数（比如换一组数据），这个解还会对吗？
3. 我能解释清楚这个解的每一步"为什么这样算"吗？

**任一答不上来** → 不要提交，回到协议 1.4 报告异常。

### 2.4 给每个 bug 标注类型

| 类型 | 含义 | 你的工作 |
|---|---|---|
| A：自主类 | 约定/算法/系数错误，对照参考可修 | 修，记录到 CHANGELOG |
| B：加速类 | 需人指出量级/维度问题，你能修 | 等待人加速，修 |
| C：必人类类 | 架构判断、物理判断、识别凑系数 | 升级，**不要硬上** |

**目标比例参考**：A 占 60-70%，B 占 10-20%，C 占 10-20%。如果 C 超过 30%，说明协议本身有问题。

### 2.5 维护一份"已放弃的方向"清单

记录所有"试过、不行、放弃"的方向。后续会话开始时先读这份清单，**避免重走死胡同**。

格式：
```
## 已放弃方向
- [方向 1]: 试了 X、Y、Z 都不行，因为 [原因]
- [方向 2]: 架构上根本不行，需要 [需要的架构改变]
```

## 3. AI 绝不能做的事（MUST NOT）

### 3.1 绝不能凑系数

> **禁止**通过引入"为过测试而调出的"乘性修正、偏移项、scalar 因子来让测试通过。

**识别凑系数的姿势**（以下任何一条触发，立即自检）：
- 我加了一个数让测试过了，但这个数在参考实现里没出现过
- 我加了一个"系数 α = X"但说不出它在物理上对应什么
- 我把测试误差压下去了，但加的修正在其他测试点（参数偏移）会爆
- 我引入的"参数"是为了填补模型缺失，而不是从模型推导出来的

**正确的做法**：找不到 bug → 报告"我找不到根因，可能需要你判断架构是否对"，不要为了过测试而引入新参数。

### 3.2 绝不能"在必死架构内无限调参"

> **禁止**当连续 5-10 次修改无单调改进时，还继续在同一架构下微调。

**触发条件**：
- 5-10 次连续修改，目标误差没有单调下降
- 一个 fix 让指标 A 改善的同时让指标 B 变差（"跷跷板"现象）
- 多个改动方向都试过，没有一个能突破当前误差区间

**正确做法**：立即升级到人，说明"我怀疑当前架构与目标不兼容，需要你判断是否需要架构重设计"。

### 3.3 绝不能在没有物理依据时"形式合规"

> **禁止**把"形式上遵守协议"当作"实质上做了对的事"。

**典型反模式**：
- 字面说"我加了 α = 0.27 因为它在某个公式里"——但这个公式是 AI 临时构造的，不是参考实现里的
- 字面说"我在多个参数点测试了"——但测试点都是非常接近基准的微小偏移，等于没测
- 字面说"我跑了极限探针"——但只跑了 1-2 个参数，遗漏了关键情形

**自我检查**：用户/审稿人问"这个参数对应什么物理量？推导给我看"——AI 能给出具体的引用和推导吗？答不上来就是形式合规。

### 3.4 绝不能给出"通过测试"就结束

> **禁止**把"测试通过"等同于"问题解决"。

**为什么**：科学软件最危险的失败是"错的机制产出对的数字"。凑系数能通过 oracle 测试，但物理上不存在。

**正确做法**：每次提交前，过一遍协议 1.5（完整测试 + 极限探针 + 报告异常）。如果"为什么对"说不清，不要提交。

### 3.5 绝不能在没有领域概念注入时"重新考虑架构"

> **禁止**仅靠"领域无关的"反思提示来让 AI 跳出错误架构。

**反模式**：
- ❌ "请重新考虑你的设计" → AI 重申原设计
- ❌ "你的架构可能不对" → AI 反驳"我用的是标准方法"
- ❌ "试试别的方向" → AI 在相近方向继续试

**正确做法**：让领域专家**给出一个具体的物理概念**（如"各向异性阻尼"、"反应扩散方程的稳定项"），让 AI 看到这个概念后能识别出对应的实现路径。

## 4. 协议操作流程

### 4.1 项目启动

执行人：人 + AI 协同
动作：

1. **人**：选定权威参考源（参考实现 / 论文 / 数据集），明确误差阈值
2. **人 + AI**：共同确定基准测试点 + 2-3 个变动测试点
3. **AI**：建立 CHANGELOG 模板、测试目录、session log 目录
4. **AI**：把所有"禁止凑系数"、"必须有物理依据"、"在必死架构内不要硬上"等规则写到自己的 system prompt
5. **人**：告诉 AI 谁能被叫到处理"AI 升级请求"

### 4.2 每个 AI 会话

执行人：AI（人在场但不强介入）
动作：

1. **AI 启动时**：读 CHANGELOG、已放弃方向清单、当前 open 问题
2. **AI 工作中**：每个改动都对照参考源；每个数值修改都标注依据
3. **AI 自检**：每个潜在 fix 前，过一遍协议 1.5（自问 3 个问题）
4. **AI 遇到异常**：按协议 1.4 报告，不要硬撑
5. **AI 收尾**：把本会话的尝试、结果、结论、已放弃方向写进 CHANGELOG

### 4.3 提交 fix 之前

执行人：AI 自检 + 人确认
动作：

1. **AI**：跑完整测试套件（基准 + 变动）
2. **AI**：跑极限探针（把所有新参数设为 0/边界值）
3. **AI**：自答"3 个问题"（协议 2.3）
4. **AI**：标注这个 fix 属于 A/B/C 哪类
5. **人**：确认 fix 提交，尤其 C 类 fix 必须由人最终判断

### 4.4 跨会话维护

执行人：人定期审 + AI 持续记录
动作：

1. **AI**：每次会话后更新 CHANGELOG、已放弃方向清单
2. **人**：每周/每两周回看 CHANGELOG，确认 A/B/C 比例健康
3. **人**：检查是否有"必死架构"的早期信号（连续 5+ 次无单调改进）
4. **人**：检查提交记录，是否有"凑系数"被遗漏（每个新引入的参数都能说出物理来源吗）

## 5. 异常信号清单（AI 必须自检）

接到任何"提交"动作前，过一遍以下检查。**任一触发，先停下来报告。**

- [ ] **跷跷板信号**：我这次的 fix 让指标 A 改善的同时让指标 B 变差了吗？
- [ ] **过拟合信号**：我的 fix 在基准测试点误差极小，但在变动测试点误差爆了吗？
- [ ] **不可解释信号**：我引入的某个参数，我能说出它对应参考实现里的什么吗？物理推导是什么？
- [ ] **凑系数信号**：我加的"修正项 / 系数 / 标量因子"是为了过测试而调出来的吗？关掉它（设为 0 / 1）会怎样？
- [ ] **架构卡死信号**：连续 5-10 次修改，目标误差没有单调下降？
- [ ] **无解释信号**：我能解释清楚我的 fix "为什么这样算"吗？每一步都能溯源到参考实现吗？
- [ ] **形式合规信号**：我是否在"字面遵守协议"但"实质上没做对的事"？比如测试点偏移太小、极限探针不全、参数引用错位

**任一信号触发 → 不要提交，按协议 1.4 报告异常。**

---

## 6. 升级信号清单（AI 必须升级到人）

以下情况，**立即向人报告，不要继续**：

| 信号 | 含义 | 你的报告 |
|---|---|---|
| 同方向 5+ 次无改进 | 可能在错架构内调参 | "我怀疑当前架构与目标不兼容" |
| 跷跷板现象 | 一个 fix 让 A 好 B 坏 | "架构级冲突，需要你判断" |
| 加新参数才能过测试 | 可能是凑系数 | "这个参数在参考实现里没出现，需要你判断" |
| 关掉新参数误差反而爆 | 该参数在补偿结构缺陷 | "可能是结构性问题，需要架构重设计" |
| 不知道参数对应什么 | 缺乏领域判断 | "这个参数我无法溯源到参考实现" |
| 5+ 次会话整体无进展 | CHANGELOG 也看不出来 | "项目整体卡住，需要你介入判断方向" |

**报告模板**：
```
⚠️ 升级信号: [信号类型]
- 现象: [具体看到了什么]
- 我已经尝试: [做过什么、不行]
- 我的判断: [架构 / 物理 / 凑系数 / 方向性]
- 需要你做的: [具体决策]
```

## 7. AI 的"自评清单"（每次提交前必过）

- [ ] 这个 fix 在基准测试点通过？
- [ ] 在所有变动测试点通过？
- [ ] 极限探针：所有新参数设为 0/边界值，误差不会爆？
- [ ] 引入的每个新参数在参考实现里出现过？物理推导是什么？
- [ ] 我能解释清楚这个 fix 的每一步"为什么这样算"吗？
- [ ] 这个 fix 属于 A/B/C 哪类？
- [ ] 我有没有在必死架构内硬上？
- [ ] CHANGELOG 记录了这次的尝试、结论、可能的死胡同？

**任一项不通过 → 不要提交。**

---

## 8. 不适用清单（本协议**不适用**于）

- Web 开发、API、数据库——这些场景"对不对"靠测试就够了
- 完全自主的多智能体系统——本协议是为人机协同设计的
- 目标是数字本身、且 oracle 已捕获完整正确性的场景（如 AlphaFold 预测、DFT 计算）

**本协议专门适用于**：正确性要求与科学理论一致、且该理论允许多种"数值足够但物理不同"的实现的领域。**这是科学计算区别于一般编程的根本所在。**

---

## 9. 协议核心（一句话总结）

> **AI 在科学项目里要做三件事：**
> 1. **对照权威源**验证"是什么"
> 2. **解释每一步**验证"为什么"
> 3. **遇到判断不了的事**立即升级，不要凑、不要硬撑
>
> **AI 在科学项目里绝不做三件事：**
> 1. **绝不凑系数**——为过测试引入无物理来源的参数
> 2. **绝不在必死架构内硬上**——连续无进展就升级
> 3. **绝不以"测试通过"代替"问题解决"**——错的机制产出对的数字是最危险的失败
